# 2.3 基于流式批处理与理论上限剪枝的RRF算法优化的实现方法

本发明提供了一种基于流式批处理与理论上限剪枝的RRF算法优化实现方法，该方法在PostgreSQL双向量列HNSW索引环境下，通过执行器层面的预计算、哈希累积、数学上界剪枝与Leader-Worker并行架构，实现双路向量检索结果的高效融合排序。

从SQL调用接口来看，RRF函数采用PostgreSQL函数重载机制设计，支持四种向量类型组合。调用格式为rrf(col1, op1, q1, col2, op2, q2, k, w1, w2, cand1, cand2)，其中col1和col2为向量列名；op1和op2为regoperator类型参数，表示距离算子的对象标识符，支持内积距离、欧氏距离、余弦距离等；q1和q2为查询向量；k为RRF平滑常数，要求大于0，默认60；w1和w2为双列权重系数，要求大于0，默认各0.5；cand1和cand2为候选集大小，要求大于0，默认各200。参数验证在vector_rrf_eval_params函数中执行，通过ExecEvalExprSwitchContext从执行器上下文中提取参数值，任一条件不满足即报错终止。

RRF计算的核心入口为vector_rrf_prepare_results函数，在CustomScan执行器的Begin阶段被调用。算法为两列分别创建流式扫描器HnswTopKStream，通过HnswTopKStreamBegin初始化，按HnswTopKStreamNextBatch以固定批次大小64获取候选结果，避免一次性加载全部候选集。批次大小由宏RRF_BATCH_SIZE定义，该值基于HNSW索引的局部性特征权衡函数调用开销与剪枝灵敏度。每批次结果存储在HnswTopKItem结构数组中，包含元组位置标识和距离值占位。

RRF分数采用哈希表累积机制。算法创建以元组位置标识为主键的HTAB哈希表，初始化时设置键大小为64位无符号整数，条目大小为RRFHashEntry结构。该结构包含组合键值、元组位置标识、两列的排序位置、距离值及累积分数。每批次结果通过rrf_hash_add_batch函数加入哈希表，遍历每个条目计算键值并查找或创建条目，首次出现则初始化排序位置和分数为0。若来自列1且该列排序位置未记录，则更新排序位置为当前深度值，按权重除以平滑常数与排序位置之和累加分数。列2同理。深度值为递增计数器，代表该结果在对应列检索流中的排序位置，确保同一行不同列的结果正确融合，每列取首次出现的最佳位置。

本发明的核心创新在于理论上限剪枝机制，由rrf_should_stop函数实现。该机制在当前批次处理完成后计算理论最高分上限，若低于已确定的Top-N最低分数则安全终止扫描。实现分为两步：第一步调用rrf_build_topk_snapshot构建Top-K快照，遍历哈希表条目复制到数组，快速排序后截取前N个，记录第N名分数作为阈值；第二步调用rrf_future_upper_bound计算理论最高分上限，假设未完整条目在下一批次获得最佳排序位置即深度值加1，计算最佳分数增量，遍历已存在但未进入Top-K的条目，取当前分数加上可能的最佳增量，更新最大上界。若阈值大于上界则触发剪枝，立即终止扫描并返回已有结果。该判定基于严格的数学上界，确保剪枝正确性，同时输出诊断信息便于性能调优。

为进一步提升性能，本发明采用Leader-Worker并行架构。Leader进程扫描列1，同时尝试启动Worker进程扫描列2。并行上下文在vector_rrf_prepare_results中创建，进入并行模式后估算共享内存需求，初始化动态共享内存，分配共享状态结构并设置表与索引的对象标识符、列位置、算子标识符、候选集与批次大小等参数，初始化自旋锁和条件变量后启动Worker。若启动失败则自动回退至串行模式。Worker从RrfParallelWorkerMain入口执行，取出共享内存参数后打开表和索引，创建流式扫描器，循环等待Leader请求，获取自旋锁检查请求编号或停止标志，收到请求后执行批次获取写入共享内存，更新深度、条目数、完成状态和就绪编号，广播条件变量通知Leader。Leader通过rrf_request_parallel_batch发送请求获取自旋锁递增编号重置条目数后广播，通过rrf_wait_parallel_batch循环检查就绪编号或完成标志，满足条件则获取条目数和完成状态，否则条件变量睡眠等待。剪枝触发或扫描完成后Leader设置停止标志通知Worker终止，等待Worker退出后销毁并行上下文，调用rrf_collect_results将哈希表结果快速排序后截取前N条返回上层执行器。

综上所述，本发明通过流式批处理降低内存占用，通过数学上界剪枝减少无效计算，通过Leader-Worker并行架构充分利用多核资源，三者在PostgreSQL执行器框架内严格协同，确保算法效率和兼容性。
